Spec written to `specs/segment-attribution-tier2/SPEC.md`. I ran the quadrant walk against the actual code first (Tier 1 endpoint at `funnelhub_core.py:2133-2450`, the decomposition math at `funnelhub_service.py:668-750`, the Trino drilldown builder, the Druid builder/drilldown/segment model, and the frontend picker), and the map is embedded in the spec.

**Core decisions:**

- **Math contract (D1):** the multi-segment axis is the **cross-product joint partition** of each segment's already-exhaustive partition (In/Out or buckets+Other). That's the only shape where the existing symmetric rate-vs-mix identity is exact, and it *represents* segment overlap as the `In A·In B` cell instead of working around it. The guarantee: per stage, rate + mix = total Δpp exactly (Trino), sketch-approximate (Druid). Interaction effects are not allocated to individual segments — per-segment numbers are explicitly a non-additive "marginal view". Capped flat at **N=2 segments / 12 cells**, with 422 (never truncation, which would break exhaustiveness). The decomposition also gains opt-in per-cell contribution output, since Tier 1 currently discards the per-cohort terms it computes.
- **Trino lane:** open funnels need **no new SQL** — `build_segment_drilldown_sql` already supports multiple membership levels (`seg_lvl_K` CTEs), so two levels give the joint cells in one query per period. Closed funnels extend Tier 1's scoped-run pattern via inclusion–exclusion (4 runs per period: total, A, B, A∩B), with clamps disclosed as `clamped: true`.
- **Druid lane (D4):** Druid can't GROUP BY theta-sketch columns and has no CASE support, so it uses **scoped-run fan-out + subtraction** (Druid runs are ~1–2s and `_apply_cohort_scope` already intersects N cohorts). Results are always flagged `approximate: true`. Degradation is **all-or-nothing per lane**: if any selected segment is Trino-only (behavioral/cohort-backed, detected via the v2-aware `effective_engines`, not the legacy empty-filters heuristic), the whole Druid lane returns `{skipped, reason}` — partial degradation would make the two lanes answer different questions.
- **API:** same endpoint, new engine-agnostic `segment_ids` (repeated, max 2) with `trino_segment_id` kept as a Tier 1 alias; new flag `FF_FUNNELHUB_SEGMENT_ATTRIBUTION_T2`, default OFF. No DB schema change.

**Top 3 unknowns that need a human decision:**

1. **Per-segment attribution semantics** — is the non-additive marginal view acceptable, or does product want Shapley-style additive allocation per segment? This changes the response shape and every piece of UI copy.
2. **Cost envelope** — confirm the N=2 / 12-cell caps against the Trino warehouse and Druid broker budgets, especially whether closed-funnel bucketed pairs (up to ~14 runs per click, cache-assisted) are acceptable.
3. **Whether sketch-approximate Druid attribution should ship at all** — mix effects are often sub-1pp, which theta-sketch noise (compounded by `Out = total − In` subtraction) can swamp; the alternative is keeping Realtime degraded until the error can be bounded.
