## What changed

The Trino funnel filter path is: client `{col, op, val}` → `from_json`/`Rule` validation → `compile_predicate_trino` → shared `emit_predicate`, with identifiers quoted via `adapter.quote_identifier` in `_render_funnel_filter`. `between` was added at each layer, following the same structure as the `in`/`not_in` peers:

**Files changed**
- `backend/dialects/predicate_compiler.py:51` — `"between"` added to `SimpleOp`; `backend/dialects/predicate_compiler.py:262-281` — new `between` branch in `emit_predicate`: accepts a two-element list/tuple (or the same comma-string fallback `IN` supports), rejects anything that isn't exactly two non-None bounds with `PredicateCompileError`, and emits each bound by delegating to the existing `>=` / `<=` branches — so literals go through `emit_literal`/the adapter (numbers unquoted via the existing ordinal-numeric heuristic, strings quote-doubled, `value_type="date"` → `DATE '...'`). Unsupported-op error message updated at line 310.
- `backend/funnelhub/predicates.py:36-49` — `"between"` added to `PredicateOp` + new `_RANGE_OPS`; `predicates.py:118-129` — `Rule` fails closed on malformed values (not exactly two non-None bounds → `PredicateAstError`, same pattern as the empty-`IN`-list peer); `predicates.py:225` — legacy alias entry (case-insensitive `BETWEEN`); `predicates.py:249` — comma-string → two-bound list coercion in `_normalize_legacy_op`.
- `backend/funnelhub/trino_sql_builder.py:1897-1900` — `_render_funnel_filter` docstring now lists `between` (no code change needed: it already routes through `from_json` → `compile_predicate_trino`, and a malformed `between` is dropped-with-warning exactly like every other malformed filter there).
- `tests/test_funnelhub_predicates.py:29-33` — `test_rule_rejects_unknown_op` used `"between"` as its unknown-op example; swapped to `"matches_glob"` so the test still guards unknown-op rejection (not weakened — the intent is preserved, the example op just graduated).
- `tests/test_funnelhub_between_filter.py` — new, 18 tests across all three layers.

**Emitted SQL shape** (parenthesized so it stays atomic under the compiler's same-field OR-grouping):
- Numeric: `("v"."amount" >= 10 AND "v"."amount" <= 20)` — numeric-string bounds `["10","20"]` also emit unquoted, matching the existing `>=`/`<=` rule.
- String: `("v"."name" >= 'O''lo' AND "v"."name" <= 'O''hi')` — quotes doubled via `adapter.string_literal`, never hand-formatted; `value_type="date"` yields `DATE '...'` bounds.
- Malformed (missing/one/three/None bounds): `PredicateCompileError` / `PredicateAstError` raised at the compiler boundary; `_render_funnel_filter` drops it with a warning like its peers — never a partial predicate.
