# Root cause

**`TrinoFunnelSQLBuilder` is a process-wide singleton that stores per-request query configuration on `self`, and concurrent funnel runs overwrite each other's fields mid-build.**

The object: the single `TrinoFunnelSQLBuilder` instance created in `make_trino_runner` at `backend/funnelhub/lane_runners.py:502` and captured by the `_run` closure. It is a singleton because the dispatcher that owns it is built once per process via `@lru_cache(maxsize=1)` on `_get_mixed_engine_dispatcher` (`backend/routes/funnelhub.py:264`), and the builder is constructed **outside** `_run`, at closure-creation time — so every Trino-lane funnel run in the process shares one builder.

The fields: `build_funnel_sql` (`backend/funnelhub/trino_sql_builder.py:549`) writes four pieces of *per-request* state onto the shared instance before building (lines 587–613):

- `self._metric_type` — customer/device/session grain selection
- `self._grain_override` — the funnel's `trino_metric_bindings.dimension` (the column inside `COUNT(DISTINCT …)`)
- `self._date_override` — the funnel's `trino_date_binding`
- `self._measure_agg` — a measure-bound metric's aggregation (SUM/AVG/COUNT(*))

These are read much later in the same call, scattered through SQL emission: `_grain()` at line 358 (chooses the count/join column), `_stage_count_expr()` at line 460 (reads `_measure_agg`), `_date_column()` at line 491. The builder's own comment at lines 337–339 claims "Builders are constructed per-request (routes + lane_runners), so this per-call state isn't shared across concurrent funnel runs" — **that invariant is false for the lane runner**, the one path every actual funnel run goes through.

# Mechanism — why only under concurrency, why Druid is fine

The dispatcher runs each lane via `asyncio.to_thread` (`backend/funnelhub/dispatcher.py:309, 333`), so two overlapping HTTP requests execute `_run` → `builder.build_funnel_sql(...)` (`lane_runners.py:589`) **in parallel OS threads on the same object**. The race window is wide: between the field assignments and the reads, the build calls `resolve_funnel_context` / `resolve_explore`, which reparse LookML from PostgreSQL per call (there is deliberately no in-process parser cache) — tens to hundreds of milliseconds of I/O with the stale-able fields exposed.

Concrete interleaving matching the bug report: user A runs a funnel configured to count distinct customers (`_grain_override = None`); user B runs a funnel bound to dimension `order_id`. A's thread sets its fields, enters the slow explore-resolution phase; B's thread then executes lines 587–613 and sets `self._grain_override = "order_id"`. When A's thread reaches `_grain(meta)` while emitting `COUNT(DISTINCT CASE WHEN … THEN <grain> END)`, it reads B's value — A's funnel returns counts at **B's configured metric dimension**. The overwrite works in both directions (B's binding can also be wiped to `None` by A, silently downgrading B to customer grain), and the same race applies to `metric_type`, the date binding, and measure aggregation.

Why each observed symptom follows:

- **Wrong grain only under load**: the fields are only stale if a second `build_funnel_sql` starts during the first one's build window. A lone re-run seconds later writes and reads its own values — correct.
- **200s, plausible data, no log errors**: the leaked grain column already passed the other request's bare-identifier validation (`_resolve_grain_override`, line 368) and typically exists on the clickstream explore, so the SQL is valid and executes cleanly — just at the wrong grain.
- **Preview always looks correct**: inspecting one request at a time means no concurrent writer; additionally, the breakdown/latency/mix-decomposition/drill endpoints construct **fresh** builders per request (`backend/routes/funnelhub_core.py:874, 1146, 2070, 2665`), so only the dispatcher-driven run path is exposed.
- **Druid unaffected**: `DruidFunnelSQLBuilder` is also a shared singleton (`lane_runners.py:74`) but is genuinely stateless — `metric_type` is threaded through as function parameters and locals (`druid_sql_builder.py:378–411`), never stored on `self`. Same lifecycle, no mutable per-call state, no race.

(Secondary, benign: the shared `_metaql_builder_cache` / `_view_obj_cache` dicts are also unsynchronized, but they're idempotent per-explore caches — worst case is redundant work, not wrong results.)

# Minimal, safe fix

Construct the builder **per run** inside `_run`, keeping the stateless `TrinoAdapter` shared. In `backend/funnelhub/lane_runners.py`, delete lines 501–506's builder creation from the closure setup and move it into `_run`:

```python
    _adapter = adapter or TrinoAdapter()

    def _run(
        stages: list[dict[str, Any]],
        segments: list[dict[str, Any]],
        funnel_type: str,
        date_range: tuple[date, date],
        user_email: str,
        runner_context: dict[str, Any] | None = None,
    ) -> LaneResult:
        # Fresh builder per run: build_funnel_sql stores per-call state
        # (_metric_type/_grain_override/_date_override/_measure_agg) on the
        # instance, and the dispatcher runs lanes on concurrent threads —
        # a shared instance leaks one run's grain into another's SQL.
        builder = TrinoFunnelSQLBuilder(
            _adapter,
            resolve_explore,
            resolve_funnel_context=resolve_funnel_context,
        )
        ...
```

This is cheap (construction is field initialization; the per-instance `_metaql_builder_cache` still deduplicates parses *within* a multi-stage run, and cross-request parser caching never existed anyway — `get_parser()` reparses per call by design) and it makes the builder's line-337 comment actually true. The longer-term cleanup — threading grain/date/measure as explicit parameters instead of instance state — is worth doing but is not needed to stop the data corruption.

# Regression test

Add to `tests/test_funnelhub_sql_builder.py` (or a lane-runner test module): drive `make_trino_runner` from two threads with a `threading.Barrier` inside the fake `resolve_explore`, guaranteeing both builds are in-flight *after* both have executed the field-assignment block — this makes the race deterministic rather than timing-dependent.

```python
def test_concurrent_runs_do_not_leak_grain_override_across_builds():
    barrier = threading.Barrier(2)
    meta = ExploreMetadata(explore_id="e1", base_table="hive.db.events", date_column="dt")

    def resolve_explore(eid):
        # Rendezvous: both build_funnel_sql calls have already written their
        # per-call fields; each now finishes its build while the other's
        # write is the last one on a shared instance.
        barrier.wait(timeout=10)
        return meta

    runner = make_trino_runner(_FakeExecutor(), resolve_explore)
    stages = [{"name": "Landing", "trino_explore_id": "e1", "trino_filters": []}]
    args = dict(segments=[], funnel_type="open",
                date_range=(date(2026, 7, 1), date(2026, 7, 2)), user_email="t@x")
    results = {}

    def run(key, ctx):
        results[key] = runner(stages=stages, runner_context={"build_only": True, **ctx}, **args)

    t_a = threading.Thread(target=run, args=("a", {"trino_metric_bindings": {"metric": "orders", "dimension": "order_id"}}))
    t_b = threading.Thread(target=run, args=("b", {}))
    t_a.start(); t_b.start(); t_a.join(10); t_b.join(10)

    assert 'DISTINCT' in results["a"].sql and '"order_id"' in results["a"].sql   # A keeps its binding
    assert '"order_id"' not in results["b"].sql                                  # B never inherits it
    assert '"customer_id"' in results["b"].sql
```

**Before the fix**: both threads write to the same instance before the barrier releases, so the last writer wins for *both* builds — regardless of scheduling order, exactly one assertion fails every run (either B's SQL counts `order_id`, or A's SQL lost its binding and counts `customer_id`). **After the fix**: each thread has its own builder, both SQLs carry their own grain, and the barrier still releases because both fresh builders call `resolve_explore`. Using `build_only: True` returns the SQL without touching a real executor, keeping the test pure-unit.
