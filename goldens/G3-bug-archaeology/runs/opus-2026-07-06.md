# Trino-lane wrong-grain under concurrency — root cause, mechanism, fix, test

## Root cause

A **single, process-wide-shared `TrinoFunnelSQLBuilder` instance stores per-request configuration (the count grain) as mutable instance attributes**, and the funnel-run path is the one path that shares that instance across concurrent requests. Two overlapping runs stomp each other's `self._grain_override` / `self._metric_type` / `self._measure_agg`, so one run emits SQL counting at the *other* run's grain.

### The precise object and lifecycle

1. **The dispatcher is a process singleton.** `backend/routes/funnelhub.py:264`:
   ```python
   @lru_cache(maxsize=1)
   def _get_mixed_engine_dispatcher():
       ...
       trino_runner = make_trino_runner(TrinoExecutor(), resolve_explore, ...)
       return MixedEngineFunnelDispatcher(druid_runner=..., trino_runner=trino_runner)
   ```
   One dispatcher for the whole process, reused on every request.

2. **The Trino runner captures ONE builder in its closure.** `backend/funnelhub/lane_runners.py:501-506` — the builder is constructed *outside* `_run` (which starts at line 508):
   ```python
   _adapter = adapter or TrinoAdapter()
   builder = TrinoFunnelSQLBuilder(_adapter, resolve_explore, resolve_funnel_context=...)  # ← once, at factory time

   def _run(...):
       ...
       sql = builder.build_funnel_sql(...)   # ← every request reuses the SAME builder
   ```
   Because the enclosing dispatcher is `lru_cache`'d, `make_trino_runner` runs exactly once ⇒ **exactly one `TrinoFunnelSQLBuilder` for the entire process**, shared by all funnel runs.

3. **`build_funnel_sql` writes request config onto `self`, then reads it much later.** `backend/funnelhub/trino_sql_builder.py:587-613`:
   ```python
   self._metric_type   = metric_type
   self._grain_override = self._resolve_grain_override(grain_override)   # line 591
   self._measure_agg   = None
   ...
   self._date_override = self._resolve_date_override(date_column_override)
   ```
   These are read far downstream — `_grain()` (line 358-366) returns `self._grain_override`, and `_stage_count_expr()` reads `self._measure_agg` — inside `_build_single_scan` / `_build_single_scan_with_segments` / `_build_chained_ctes` (lines 645-658), i.e. *after* the writes, across many bytecodes.

The code even documents the invariant it relies on — and that invariant is false for this path (`trino_sql_builder.py:336-338`):
> *"Builders are constructed per-request (routes + lane_runners), so this per-call state isn't shared across concurrent funnel runs."*

Routes **do** construct per-request (`funnelhub_core.py:874/1146/2070/2665`, `funnelhub_segments.py:419/485` — a fresh builder inside each handler). **`lane_runners.py` does not.** The comment is half-right, and the funnel-run path is the exception.

## Why only under concurrency

The dispatcher runs each lane on a worker thread and gathers them (`dispatcher.py:387` `asyncio.gather(...)` over `asyncio.to_thread(...)` at lines 309/333/368). Two overlapping funnel runs → two `to_thread` workers → two threads calling `build_funnel_sql` **on the same instance**, truly interleaved. The write→read span (line 591 → deep inside `_build_single_scan`) is not atomic; the GIL yields between bytecodes. Interleaving:

```
Thread A (funnel 1, grain = customer_id):  self._grain_override = "customer_id"   (591)
Thread B (funnel 2, grain = order_id):     self._grain_override = "order_id"      (591)  ← overwrites
Thread A: ... _grain() reads self._grain_override → "order_id"                    ← wrong grain
```
Run A emits `COUNT(DISTINCT order_id)` — funnel 2's configured metric dimension — exactly the report ("counts match a different funnel's configured metric dimension"). Run alone, seconds later, there's no interleaving, so it's correct. It also fires for a single user issuing two quick runs (two overlapping `to_thread` workers).

## Why the preview SQL always looks correct

`POST /query/preview-sql/v2` builds via a **route-handler-local** builder (`funnelhub_core.py`), a fresh instance per request — never shared. Inspected one at a time, it's always right. Only the *run* path (dispatcher singleton → shared builder) corrupts.

## Why Druid is unaffected

`make_druid_runner` also holds a single `DruidFunnelSQLBuilder` (`lane_runners.py:74`), but that builder is **stateless w.r.t. request config**: `metric_type` is threaded as a *method argument* (`_metric_expr_open(self, metric_type, stage_where)`, `druid_sql_builder.py:98/104`), living on each thread's own call stack. Its only `self.` attributes are immutable constants (`_METRIC_FIELD`, quoting helpers). A shared instance with no shared mutable state is thread-safe — hence Druid numbers are always correct.

## The fix (minimal, safe)

Restore the invariant the code already assumes: **construct the Trino builder per run instead of once per process.** In `backend/funnelhub/lane_runners.py`, move the construction from the factory body into `_run`:

```python
def make_trino_runner(trino_executor, resolve_explore, adapter=None,
                      drift_resolver=None, resolve_funnel_context=None):
    _adapter = adapter or TrinoAdapter()
    # NOTE: builder is per-run, not closure-captured — its build_* methods stash
    # per-call config (grain/metric/measure) on self, so a shared instance
    # corrupts under the dispatcher's parallel to_thread lanes.

    def _run(stages, segments, funnel_type, date_range, user_email, runner_context=None):
        builder = TrinoFunnelSQLBuilder(          # ← construct here, per invocation
            _adapter, resolve_explore, resolve_funnel_context=resolve_funnel_context,
        )
        ...
        sql = builder.build_funnel_sql(...)
```

`TrinoFunnelSQLBuilder.__init__` only initializes a few empty dicts and stores callables — construction is trivially cheap; the expensive LookML parsing lives behind the shared `resolve_funnel_context` callable, which is unaffected. The builder's internal `_metaql_builder_cache` / `_view_obj_cache` are per-explore parse caches meant to avoid reparsing *within one multi-stage build* — per-run instantiation preserves exactly that scope; only cross-request reuse of those caches is lost, which is correctness-critical to drop anyway since they sit on the same mutable object.

**Why this over alternatives:** wrapping `build_funnel_sql` in a `threading.Lock` would serialize all Trino SQL builds process-wide (throughput hit) and still leaves a foot-gun. Threading the config through as parameters/a context object (removing `self._grain_override` et al.) is the more thorough fix but touches `_grain`, `_stage_count_expr`, and every deep call site — larger blast radius. Per-run construction is a 3-line move that directly reinstates the documented "per-request builder" contract, matching what the route handlers already do.

## Regression test

Add to `tests/test_funnelhub_sql_builder.py` (or a `test_funnelhub_lane_runners.py`) a **concurrency test at the runner level**, made deterministic with a barrier so it fails reliably before the fix rather than flaking:

```python
def test_trino_runner_isolates_grain_across_concurrent_runs():
    # A resolve_explore that blocks after each thread has set its own grain,
    # forcing the interleaving deterministically (both threads pass line 591
    # before either reaches the _grain() read inside _collect_explore_ctes).
    barrier = threading.Barrier(2)
    def resolve_explore(eid):
        barrier.wait(timeout=5)          # release only once BOTH have set grain
        return _fake_meta(eid)           # single-view, customer-grain explore

    runner = make_trino_runner(FakeTrinoExecutor(), resolve_explore)

    results = {}
    def run(label, dim):
        results[label] = runner(
            stages=[_stage("s1", explore="e1")], segments=[], funnel_type="open",
            date_range=(D0, D1), user_email="t@x.com",
            runner_context={"trino_metric_bindings": {"dimension": dim}},
        ).sql

    a = threading.Thread(target=run, args=("A", "customer_id"))
    b = threading.Thread(target=run, args=("B", "order_id"))
    a.start(); b.start(); a.join(); b.join()

    # Each run's SQL must count on ITS OWN grain, never the other's.
    assert "COUNT(DISTINCT customer_id)" in results["A"] and "order_id" not in results["A"]
    assert "COUNT(DISTINCT order_id)"   in results["B"] and "customer_id" not in results["B"]
```

- **Before the fix:** the two threads share one builder; the barrier guarantees B overwrites `self._grain_override="order_id"` before A reads it, so A's SQL contains `order_id` → assertion fails (reproducing the production symptom exactly).
- **After the fix:** each `_run` gets its own builder, `self._grain_override` is thread-private, both assertions pass.

A companion **negative/control test** asserting the Druid runner stays correct under the same concurrent harness (it will already pass, before and after) documents the asymmetry and guards against a future refactor moving Druid config onto `self`.
