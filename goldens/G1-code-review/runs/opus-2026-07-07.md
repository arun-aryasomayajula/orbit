# FunnelHub Subsystem — Security & Correctness Review

## Critical

### 1. SQL injection — `DATE '`/`CURRENT_DATE`-prefixed filter values pass through unescaped
**`backend/dialects/predicate_compiler.py:154-157`** (sink), reached from **`backend/funnelhub/trino_sql_builder.py:1936`** via **`backend/funnelhub/predicates.py:361`**.

`_render_funnel_filter` compiles client funnel/stage/drill filter values with `value_type=None`; `emit_literal` returns *any* string starting (case-insensitively) with `DATE '`, `TIMESTAMP '`, `CURRENT_DATE`, or `CURRENT_TIMESTAMP` verbatim, with no escaping.

**Failure scenario:** an authenticated user runs a Trino-lane funnel (e.g. `POST /query/execute`, or a v2 drill with `extra_filters`) with a stage filter `{"col":"event_category","op":"=","val":"DATE '1' OR 1=1 --"}`. The builder emits `"..._cte"."event_category" = DATE '1' OR 1=1 --`. The attacker controls everything after the prefix, so `DATE '1' OR customer_id IN (SELECT ...)` executes arbitrary SQL on the warehouse. Works inside `IN(...)` too (`emit_in_list` calls `emit_literal` per element). Not feature-flag gated — always live.

### 2. SQL injection — `date_literal` does not escape values on date-typed dimensions
**`backend/dialects/trino/adapter.py:123-124`** (`return f"DATE '{iso_date}'"`), reached from **`backend/funnelhub/trino_sql_builder.py:3663`** (`_compile_predicate` supplies `resolve_field_type`, used by `_build_segment_ctes` at ~2818).

When a filter targets a `type: date` LookML dimension, `emit_literal` (`predicate_compiler.py:171-172`) routes to `date_literal`, which f-strings the raw value with no quote-doubling.

**Failure scenario:** a segment/stage filter on a date dimension with `{"col":"dl_last_updated","op":"=","val":"2020-01-01' OR '1'='1"}` emits `"..._cte"."dl_last_updated" = DATE '2020-01-01' OR '1'='1'` — the embedded quote breaks out of the literal. Every date-typed column filter is an injection point. (Fix both: restrict the prefix passthrough to internally-generated literals and validate/escape `date_literal` input against an ISO-date regex.)

## Major

### 3. Cross-tenant view resolution — `_find_view_by_name` ignores the explore's own scope
**`backend/funnelhub/explore_resolver.py:292-302`**, called at `:225`.

`resolve_explore_with_context` loads `explore_row` carrying `use_case`/`vertical_id`/`scope`, then resolves the referenced view by **name only**, ordered globally `scope = 'use_case' DESC, updated_at DESC` — the explore's tenant is never passed in. The resolved view's `sql_table_name` becomes the funnel's `base_table`.

**Failure scenario:** tenant A's explore references view `orders`; tenant B independently defines a `use_case`-scoped view named `orders` pointing at B's warehouse table, updated more recently. Running A's Trino funnel resolves `base_table` from B's view → A's funnel silently queries B's physical table (cross-tenant metadata bleed + wrong results). LookML name collisions like this are routine.

### 4. Authorization bypass cluster — persisted-resource endpoints skip the `_authorize_funnel`/`_check_vertical_role` their siblings apply
All gated behind `FF_FUNNELHUB_RBAC` (default OFF today), so **latent now but live the moment an operator enables RBAC** — which the platform is explicitly built to support (grants get seeded, flag flipped). The danger is the inconsistency: enabling RBAC locks down `GET /funnels/{id}` but leaves these open.

- **`backend/routes/funnelhub_core.py:296-317`** `GET /chart/{chart_id}/data` — loads any funnel by id and executes it on Druid, returning live conversion data, with no authorize call (contrast `get_funnel` at `:115`).
- **`backend/routes/funnelhub_core.py:2689`** `POST /query/trino-drill-breakdown` — the only v2 funnel-id endpoint missing `_authorize_funnel` (all siblings at 519/688/793/1032/1213/1470/1680/1972/2190/3352 have it); also resolves arbitrary `segment_ids`.
- **`backend/routes/funnelhub_segments.py:352-372`** `GET /segments/{id}/preview` — runs a Druid preview of any segment with no `_check_vertical_role` (contrast `get_segment` at `:193`).
- **`backend/routes/funnelhub_core.py:286-293` / `329-357`** `GET /chart/{id}` and `/form_data` — return any funnel's full definition (stages, filters, wiring) with no authorize.
- **`backend/routes/funnelhub_insights.py:130-136` / `192-202`** `GET /insights/{id}` and `/snapshots` — skip the accessibility gate `list_insights` enforces; snapshots are materialized historical results.

**Failure scenario (RBAC on):** a user holding `explorer` only on "Recharge" calls `GET /api/funnelhub/chart/8/data` for a "Lending" funnel and reads Lending's stage-by-stage numbers; ids are enumerable and `_get_funnel_by_any_id` also resolves `source_chart_id`.

### 5. Blocking DB IO inside `async def` handlers stalls the event loop
**`backend/routes/funnelhub_core.py:516, 527, 685, 713, 790, 1029, 1467/1477, 1677/1687, 1969/1994, 2189, 2689`** — every async v2 endpoint calls `_get_dao().get_funnel(...)` / `.get_segment(...)` directly on the loop. Confirmed blocking: `funnelhub_dao.py:566-567` uses the synchronous `get_db_cursor()` pool. Only the downstream Trino/Druid work is wrapped in `asyncio.to_thread`.

**Failure scenario:** under DB latency or pool saturation, each mixed-engine request performs 1 + N (per segment) synchronous round-trips on the event loop before dispatching lanes, blocking *all* concurrent requests process-wide — the exact guardrail this codebase documents as non-negotiable.

### 6. Fire-and-forget Trino lane + illusory `wait_for` timeout risks default-threadpool exhaustion
**`backend/funnelhub/dispatcher.py:311-317`** (comment wrongly claims `wait_for` cancels the `to_thread` worker), **`:407-413`** (shielded background task), **`:161`** (`trino_timeout_seconds=1800`).

`asyncio.wait_for` never kills the `to_thread` worker — the blocking query runs to completion regardless. The Trino lane is deliberately shielded so it survives client cancellation. The only real backstop is the executor's own request timeout, which is not set/enforced at the lane call sites (`lane_runners.py:168, 631`).

**Failure scenario:** ~16-32 users open mixed-engine funnels and navigate away mid-run; each pins default-`ThreadPoolExecutor` threads (`min(32, cpu+4)`) that keep executing until the engine's own timeout (if any) fires. Once the pool saturates, every `asyncio.to_thread` call process-wide — including this module's own cache read/write at `:269/:382` — queues behind them, producing a process-wide latency spike/apparent hang.

### 7. `_resolve_global_filters` scopes on client-supplied `vertical_id`/`product_id`
**`backend/services/funnelhub_service.py:117-118`.** The vertical/product row-scoping filters merged into the query WHERE are keyed off the request payload, never reconciled with a persisted funnel (`get_funnel` isn't called on the v1 `execute_query`/`preview_sql`/`execute_breakdown`/`execute_drilldown` paths). `get_vertical_filters(None)` returns `[]`.

**Failure scenario:** a caller omits/changes `vertical_id` in the payload; the mandatory vertical-scoping predicate is silently dropped and the Druid query scans outside the intended vertical. `preview_sql` returns the generated SQL, making the missing scope directly observable. (Consistent with the "open compute plane" design while RBAC is off, but it means these filters can never be relied on for isolation.)

## Minor

### 8. `update_segment` authorizes the client-supplied vertical, not the persisted row's
**`backend/routes/funnelhub_segments.py:255-256`** checks `request.vertical_id` (the incoming target), then writes it back as the new `vertical_id`. The vertical gate is on the wrong value; `_ensure_owner` (creator check) is the only real guard. With RBAC on, a creator who lost `developer` on the segment's current vertical can still edit/relocate it by passing a vertical they do hold.

### 9. `_check_platform_admin` is a no-op while call sites claim it gates admins
**`backend/routes/funnelhub.py:247-251`** returns unconditionally; `create/update/delete/batch_delete_segment` (`funnelhub_segments.py:211,258,309,533`) comment that global segments "require platform admin." Effect: any authenticated user can create/edit/delete `is_global=True` segments that are visible/usable across every vertical. Either enforce `is_admin(...)` or delete the misleading comments.

### 10. `time_ranges.py` unguarded `date.fromisoformat` crashes the scheduler
**`backend/funnelhub/time_ranges.py:72-75`.** `_ABS_TO` regex matches calendar-invalid dates (`2026-13-01`); unlike the ISO-datetime branch below it, this branch has no `try/except ValueError: return None`.

**Failure scenario:** an insight stored with `time_range = "2026-13-01 to 2026-12-01"` makes `resolve_time_range_to_dates` raise instead of returning `None`, crashing that insight's scheduled refresh.

### 11. `serialization.py` does not sanitize NaN/NaT nested in list/dict values
**`backend/funnelhub/serialization.py:56-58`** only inspects top-level scalars; `list`/`dict` values are excluded and never recursed. A row column that is a JSON array/object containing `float('nan')` survives and trips FastAPI's encoder with the exact "Out of range float values are not JSON compliant" 500 this module exists to prevent.

### 12. Insight alert/schedule reads leak recipient emails
**`backend/routes/funnelhub_insights.py:243-255, 352-358`** only check the insight exists, then return schedule/alert data including `recipient` email addresses (`_serialize_alert`). Any authenticated user enumerating `insight_id` harvests alert recipients and cron schedules; the write siblings are correctly owner-gated.

## Overall assessment

The subsystem is competently built and shows real security maturity in its recently-hardened areas — identifier interpolation is uniformly routed through `quote_identifier`/allowlists, Druid datasources are allowlisted, every `REGEXP_LIKE` site is ReDoS-guarded, DAO queries are properly parameterized, and the mixed-engine dispatcher's parallel-lane structure (per-lane timeouts, failure isolation, `to_thread` wrapping of heavy IO, per-call SQL builders) reflects lessons from the documented June-2026 review. The serious gaps are concentrated in two seams that the hardening passes missed: **value literalization** — where the shared `predicate_compiler`/`adapter` emitters that `predicates.py` trusts as the escaping boundary have two unescaped escape hatches, yielding a real, always-live SQL-injection class from ordinary client filter values (findings 1-2, the top priority) — and **tenant isolation**, which is coherent on the v2 write path but has a cross-tenant view-resolution bug (3) plus a cluster of read/execute endpoints that will silently stay open when `FF_FUNNELHUB_RBAC` is enabled (4). Rounding it out are an event-loop-blocking pattern in the async handlers (5) and a threadpool-exhaustion risk in the fire-and-forget dispatcher (6). None of the authz issues are exploitable *today* (RBAC default-off, FunnelHub intentionally open), but the injection findings are, and they should be fixed with regression tests before anything else.
