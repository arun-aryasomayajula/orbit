# FunnelHub Backend Review — Findings

**Scope-wide caveat on the authorization findings:** `FF_FUNNELHUB_RBAC` defaults **OFF**, and while off `_check_vertical_role`, `_authorize_funnel`, `_check_platform_admin`, and `_accessible_*` are all no-ops (`backend/routes/funnelhub.py:222-291`) — FunnelHub is intentionally open to any authenticated user today. The authz findings below are the defects that surface **the moment an operator flips the flag on** (the documented deployment path, per project memory: seed grants → enable flag). They are real because the v2 query surface *does* enforce the guard, so these endpoints are an inconsistent gap in the protection the flag is supposed to provide — not theoretical hardening. The correctness/async/DoS findings are active **regardless** of the flag.

## Critical

**1. Unbounded fire-and-forget Trino tasks strand the shared thread pool → process-wide stall.** `backend/funnelhub/dispatcher.py:65-71, 407-422`. Every Trino lane is launched as a shielded background task held in a module-level `set` with no concurrency cap and no dedup; on client disconnect or caller `wait_for` timeout the shield keeps the uncancellable 30-minute `asyncio.to_thread` worker running. Failure scenario: one `analyze/v2` request fans out up to 24 cells (`funnelhub_core.py`), each with a 200s `wait_for` over a lane whose Trino task is shielded for 1800s; when the cells time out, ~24 slow Trino threads survive and exceed the default executor pool (`min(32, cpu+4)` ≈ 12 on an 8-core box), so *every* `to_thread` in the process — cache reads/writes, the hourly cache GC, other requests' DB calls — queues behind 30-minute queries and the platform appears hung. The inline comment at `dispatcher.py:311-313` ("`asyncio.wait_for` cancels the underlying `to_thread` worker") is factually wrong and contradicts the repo's own guardrail; the worker runs to completion.

**2. Cross-tenant funnel *execution* endpoints skip `_authorize_funnel` entirely (flag-on IDOR).** `backend/routes/funnelhub_core.py:296-314` (`GET /chart/{chart_id}/data`) and `2616-3174` (`POST /query/trino-drill-breakdown`, loads funnel at `2689`). Both load a funnel by id and execute it against Druid/Trino, returning row data, with no vertical check — while the sibling v2 executors call `_authorize_funnel` at lines 519/793/etc. and the drill's own picker endpoint `get_trino_dimensions` authorizes at 3352. Failure scenario (flag on): a user scoped to vertical A iterates `chart_id`/`funnel_id` and pulls every other vertical's funnel results and drill breakdowns. The entire **v1 body-shaped query surface** (`funnelhub_core.py:365, 376, 1372, 1398, 2453, 2481, 2509, 2550, 3588, 3636`) has the same gap — no tenant gate at all, so the flag protects nothing on that surface.

## Major

**3. Read-only cross-tenant leaks (flag-on).** `GET /chart/{chart_id}` (`funnelhub_core.py:286-293`) and `GET /form_data` (`329-357`) return the full funnel definition (stages, filters, owner email) with no vertical check; `GET /segments/{id}/preview` (`funnelhub_segments.py:352-372`) executes a Druid cohort scan with none, while its sibling `GET /segments/{id}` (`183-199`) does check; `GET /explores/{explore_id}/grain-dimensions` (`funnelhub_core.py:3308-3309`) discloses any explore's schema. Failure scenario: user blocked from vertical X reads segment 42's definition + matched counts via the `/preview` route it wasn't gated on.

**4. Segment preview/values endpoints run warehouse queries on a client-supplied `explore_id`/`datasource` with zero authz + allowlist bypass.** `funnelhub_segments.py:340-349, 375-449, 460-516`; `SegmentPreviewRequest` (`routes/models.py:896-905`) has `datasource: str = "pulse-production"` with **no** `check_datasource` validator (unlike `FunnelDefinitionCreateRequest` at 544 and `FunnelQueryRequest`), and `segment_preview._quote_datasource` (`segment_preview.py:46-52`) validates charset only — no `ALLOWED_DATASOURCES` allowlist, unlike `DruidFunnelSQLBuilder._quote_table` (`druid_sql_builder.py:87-93`). Failure scenario: any authenticated user posts an arbitrary MetaQL `explore_id` + column to `/segments/preview/trino` or `/trino/values` and enumerates distinct values (potential PII) of any vertically-scoped explore; or posts `datasource: "some_other_internal_ds"` and scans a datasource FunnelHub was never meant to reach.

**5. Insight write authorization is a permanent no-op.** `funnelhub_insights.py:107, 144, 180` call `_check_vertical_role(..., None, 'developer')` — passing literal `None`, which the helper early-returns on (`funnelhub.py:224`), so even with the flag on the insight's real vertical (derivable from its `funnel_ids`) is never resolved or enforced; an explorer-only user still creates/updates/deletes insights. Related: attached `funnel_ids`/`segment_ids` are inserted with `ON CONFLICT DO NOTHING` and no existence/access check (`funnelhub_insights.py:123-124, 166-167`).

**6. Orphaned insight-alert edits bypass ownership.** `funnelhub_insights.py:411-436` + `funnelhub.py:196-197`. Alert update/delete resolve the parent via `get_insight()`, which filters `is_active = TRUE` and returns `None` for a soft-deleted insight; `delete_insight` does not cascade to `fh_insight_alerts`, and `_ensure_owner(None, ...)` treats `None` as "allow". Failure scenario: user A creates an insight+alert then deletes the insight; the alert row survives, and any authenticated user holding the alert UUID can `PUT /alerts/{id}` to repoint `recipient` (exfiltrating funnel metrics by email) or `DELETE` it — ownership is never verified.

**7. `PUT /funnels/{id}` lets the payload move a funnel into any vertical.** `funnelhub_core.py:165-186`. Authorization checks the *existing* row's vertical (165), but the new `payload["vertical_id"]` is only existence-checked (173), never role-checked — asymmetric with `create_funnel`, which role-checks the target vertical at line 123. Failure scenario (flag on): a developer on vertical A re-PUTs their funnel with `vertical_id=B`, planting content into vertical B where they hold no role and where B's explorers will now read it.

**8. `GET /funnels` with `vertical_id` omitted returns every vertical's funnels.** `funnelhub_core.py:96-99` + `funnelhub_dao.py:500-507`. `_check_vertical_role(None,...)` is a no-op for `None`, and `list_funnels` adds the `f.vertical_id = %s` clause only when `vertical_id is not None`. Failure scenario (flag on): an attacker simply omits the query param and the DAO returns all rows across all tenants.

**9. Blocking IO on the event loop in the async v2 handlers.** `funnelhub_core.py:516, 527, 685, 713, 790, 1029, 1210, 1467, 1477, 1677, 1687, 1969, 1994, 2187, 2291, 2689, 2860, 3008` call `dao.get_funnel`/`get_segment`/etc. — all `with get_db_cursor()` (blocking psycopg2) — directly inside `async def`, un-wrapped, even though the same handlers wrap the heavy Trino/cache calls in `asyncio.to_thread`. Worse, `execute_trino_drill_breakdown` calls `resolver._resolve_level` at `2895`, which for a regex segment fires a synchronous Druid `GROUP BY` HTTP round-trip on the loop (`funnelhub_drilldown.py:338, 378-397`) — exactly the full-scan contention class behind the 2026-06-25 timeout incident. Failure scenario: pool exhaustion (max 20) or one slow Druid scan freezes the loop, stalling all concurrent requests including health checks.

**10. Druid regex ReDoS guard is bypassable → shared-cluster DoS (always active).** `backend/funnelhub/filter_translator.py:87`, reaching `REGEXP_LIKE` at `151-156`. `_REDOS_PATTERN = re.compile(r'\([^)]*\)[+*]\??')` only catches a single non-nested group followed by a bare `+`/`*`; interval-quantified `(a+){2,}` (next char is `{`) and nested `((a+))+` both pass, and `re.compile` checks only syntax, not cost. Failure scenario: any authenticated user creates a Druid filter with `op:"REGEX"`, `val:"(a+){2,}$"` (≤200 chars); against long `event_label` values Druid's Java regex backtracks catastrophically, burning shared cluster CPU.

**11. Trino result cache key omits the row `limit` → truncated data served as complete for 24h (cross-confirmed).** `backend/funnelhub/trino_funnel_cache.py:96-130` (`CacheKeyInputs` has no limit field); breakdown call site `funnelhub_core.py:851-860` (limit at 887 but absent from key) and drill site `3057-3067` (`request.limit` shapes SQL at 3026, absent from key). Failure scenario: user A runs a breakdown/drill with `limit=50`; user B runs the identical query with `limit=2000` and silently receives A's 50-row payload — plus the cached body carries A's `truncated` flag. (`funnelhub_core.py:3146` also computes `truncated` against `request.limit` while bucketed drills raised the SQL limit to `max(request.limit, 2000)`, mislabeling non-truncated results.)

**12. Unsanitized NaN in Trino rows 500s the whole mixed-engine response and silently breaks the cache write.** `lane_runners.py:682-695`, `dispatcher.py:376-389`, returned raw at `funnelhub_core.py:642-647`. `pandas.read_sql_query` converts SQL NULLs in numeric columns to `float('nan')`; the v2 execute path returns `envelope.trino.__dict__` without `serialization.sanitize_records` (applied on drill/breakdown paths but not here), so Starlette's `json.dumps(allow_nan=False)` raises `ValueError` → the entire response 500s, discarding the healthy Druid lane. The same NaN silently invalidates the JSON written to `fh_funnel_cache` (`trino_funnel_cache.py:316`), swallowed as a WARN, so that funnel pays full Trino latency forever.

**13. `_extract_trino_counts` crashes the lane on a NULL stage value.** `lane_runners.py:750-754` calls `int(value)` unguarded, whereas its bucketed twin (`791-795`) wraps it in `try/except (TypeError, ValueError)`. Failure scenario: a measure-bound (SUM/AVG) metric over an empty stage returns SQL NULL → pandas NaN → `int(nan)` raises `ValueError`, turning the whole Trino lane into an error instead of showing 0.

**14. Regex stage filters are silently dropped on the Trino common path.** `trino_sql_builder.py:3599-3613` + `_MAP_FUNNEL_OP_TO_METAQL` (`100-117`). MetaQL's `build_filter_condition` (`query_builder/filter.py:737`) returns `None` for unmapped ops (no regex branch); `_try_compile_via_metaql` does `if frag: fragments.append(...)` and still returns a non-`None` string, which blocks the legacy fallback. Failure scenario: a stage with `[{event ==}, {screen regex '^/recharge'}]` compiles to `event = 'pay'` only — the regex vanishes and the stage count inflates (or compiles to `TRUE` if regex was the only filter), while the Druid lane applies it correctly, so the two lanes silently disagree. On paths that skip MetaQL (`_build_single_scan_with_segments`, `_build_chained_ctes`) the same regex op raises in `predicates.py:94` and fails the whole run.

**15. `trino_date_binding` is applied to the funnel run but ignored by breakdown/drill/latency/cohort-count.** `trino_sql_builder.py:593-657` (`build_funnel_sql` accepts `date_column_override`), but `build_breakdown_sql` (706), `build_segment_drilldown_sql` (976), `build_latency_sql` (1939), and `build_segment_cohort_count_sql` (888) take no date param and never set `_date_override`, so `_date_column` (537-538) falls back to `meta.date_column`. Failure scenario: a funnel bound to `txn_settled_date` filters the diagram on that column but every breakdown/drill/latency filters and buckets on the explore's default `date_partition` — a different row set, silently non-reconcilable. (Asymmetric with the grain override, which *was* threaded through all four builders.)

**16. `_extract_bare_column` mis-extracts a single column from complex measure SQL.** `trino_sql_builder.py:437-443`. `re.search(r"\$\{([^}]+)\}...")` matches the *first* `${…}` and never verifies it spans the whole expression, so `${amount} * ${fx_rate}` → `"amount"` and `COALESCE(${amt},0)` → `"amt"`. Failure scenario: a Trino metric bound to `type: sum, sql: ${amount} * ${fx_rate}` emits `SUM(... "amount" ...)` — un-converted sums instead of falling back to grain with a warning (the guardrail's "no silent grain shift" class). Not injection (still passes `_GRAIN_IDENT_RE`), but wrong numbers.

**17. `_render_funnel_filter` silently drops bare-column non-regex drill filters.** `trino_sql_builder.py:1927-1931`, consumed at 1230-1237 / 1696-1703. `from_json([f])` raises on a dotless `col` (`predicates.py:68-72, 254`); the exception is swallowed and the filter dropped. v2 cohort-scope segments produce exactly these bare cols (`segment_definition.py:696-697`), and `_build_segment_ctes` qualifies them before `from_json` (`2813-2817`) but `_render_funnel_filter` doesn't. Failure scenario: drilling by a segment with a `status == 'active'` equality residual runs without that restriction — drill totals include inactive customers and exceed the cohort, no visible error.

## Minor

- **`timezone` interpolated into Druid SQL without quote-escaping** (`druid_sql_builder.py:434, 581`) — not currently reachable (all callers validate against `available_timezones()` and the persisted runtime is built server-side), but zero defense-in-depth; escape it or validate in the builder.
- **Unbounded in-memory `_TRINO_VALUES_CACHE`** keyed on free-text `search` with lazy-only TTL, no size cap (`funnelhub_segments.py:456-515`) — slow memory-growth DoS.
- **No single-flight on the Trino lane** (`dispatcher.py:264-390`) — N identical concurrent runs all miss and all fire full warehouse queries (the sibling `query_cache.py` dedups; `fh_funnel_cache` doesn't); compounds finding 1.
- **`P3M`/`P1Y` bucket keys never canonicalized** (`lane_runners.py:356-367`) — Druid (UTC instants) and Trino (local dates) rows render as separate rows for quarter/year granularities; the "would have 422'd upstream" comment is stale.
- **Timeout-less lane gathers** in `execute_latency_v2` (`funnelhub_core.py:1357`) and `mix_decomposition_v2` (`2349-2354, 2445`) — hand-rolled `asyncio.gather` with no per-lane `wait_for`, unlike the dispatcher-migrated breakdown path.
- **Unbounded `segment_ids`** on `execute/v2` and `preview-sql/v2` (`funnelhub_core.py:458, 659`) — no `max_length` (vs 5/6 elsewhere); N sequential blocking DB reads + N cohort JOINs.
- **Unknown `funnel_type` silently coerces to "open"** (`dispatcher.py:215-218`) — a corrupt/novel value produces independent-count (open) instead of cumulative (closed) numbers, and the coerced value is hashed into the cache key.
- **LookML explore edits neither change the cache key nor invalidate it** (`trino_funnel_cache.py:133-144`; invalidation fires only on funnel PUT/DELETE) — editing a dimension's SQL or grain tag serves pre-edit numbers for up to 24h.
- **Fixed 24h TTL contradicts the T-1 freshness model** (`trino_funnel_cache.py:44-49, 295`) — a row written at 23:50 serves ~24h past the midnight-UTC refresh that supersedes it.
- **`sanitize_records` mangles a single dict/str input** into a list of keys/chars (`serialization.py:45-46`) — needs a `dict`/`str` check before the generic-iterable branch.
- **Funnel-run segment JOINs assume the stage explore's grain** (`trino_sql_builder.py:3293-3299, 3421-3426`) — a `device_grain`/`customer_id` mismatch references a column the segment CTE never projects → `COLUMN_NOT_FOUND`; the drill paths already solved this with `_segment_cohort_grain_col`.

## Overall assessment

The subsystem is functionally mature and shows real security discipline where it counts most — the Trino SQL builder's identifier quoting and literal escaping are thorough and I found no injection path, per-request builders correctly avoid the shared-singleton race the guardrails warn about, and the v2 query surface consistently authorizes against the persisted funnel's own vertical. But the health is uneven along two seams. First, **authorization is bifurcated**: the v2 surface enforces the RBAC contract while the legacy v1 query surface, the `/chart/*` and `/form_data` compatibility endpoints, insight writes, and segment-preview/values endpoints do not — so the day an operator turns `FF_FUNNELHUB_RBAC` on (the intended rollout), several cross-tenant read and execute paths remain wide open, defeating the flag's purpose. Second, and independent of any flag, the **Trino lane's operational and correctness edges are fragile**: the unbounded fire-and-forget shield can starve the whole process's thread pool, NaN handling is missing on the one row-emitting path that isn't sanitized (500-ing healthy responses and silently poisoning the cache), the cache key omits `limit`, and at least four builder paths silently diverge from the funnel diagram (dropped regex/bare filters, ignored date binding, mis-extracted measure columns) producing plausible-but-wrong numbers with no user-visible error — the most dangerous failure mode for an analytics tool. None of these are exotic; they cluster in the places where the newer v2/mixed-engine work grew past the older code without back-filling the guards and sanitizers the rest of the module already established. Prioritize finding 1 (availability, always active) and findings 11–17 (silent wrong data, always active), then close the authz gaps as a set before enabling RBAC.
